상태: Accepted
날짜: 2026-07-19
================================================================================

Context
--------------------------------------------------------------------------------
Phase 6b.0 added vehicles.operator_id. Maintenance records and per-vehicle
schedules still listed the whole shared DB, so a second tenant with Ops ON
could see Kovegas fleet history.


Decision
--------------------------------------------------------------------------------
1. Migration: 20260719130000_saas_operator_id_vehicle_maintenance.sql
   - vehicle_maintenance.operator_id NOT NULL (default Kovegas)
   - vehicle_maintenance_schedules.operator_id NOT NULL (default Kovegas)
   - Backfill from vehicles.operator_id when vehicle_id present
   - No schema rename/move; catalog stays shared platform reference

2. API list/create stamp:
   - GET/POST /api/vehicle-maintenance?operatorId=
   - GET/PUT /api/vehicle-maintenance/schedules?operatorId=
   - Insert stamps from vehicle.operator_id (fallback request operator / Kovegas)
   - sync from company_expenses stamps via vehicle map + operator scope

3. Clients pass active operatorId on maintenance fetches.


Non-goals
--------------------------------------------------------------------------------
- vehicle_maintenance_catalog tenant split (shared codes / FK)
- company_expenses.operator_id (later 6b.x)
- ops_* schema move
- Booking / payment / checkout changes
- Full Ops RLS rewrite
