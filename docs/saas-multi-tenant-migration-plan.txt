================================================================================
0. 결론 (먼저 읽기)
================================================================================

권장 전략
--------------------------------------------------------------------------------
1) 테넌시: Shared Database + tenant_id (Row-Level Isolation)
   - 초기·중기 최적. Schema-per-tenant / DB-per-tenant는 나중에 엔터프라이즈만.

2) 도메인: v2 greenfield 모델을 그대로 쓰되, 모든 비즈니스 테이블에
   operator_id (tenant) 를 최상위 키로 둔다.

3) 마이그레이션: Big-bang 금지.
   - Phase A: 현재 DB에 operator_id 도입 (Kovegas = tenant #1)
   - Phase B: v2 가격/재고/Offer 엔진을 tenant-aware로 신설 (dual-write)
   - Phase C: 신규 공급자 onboarding 오픈
   - Phase D: 레거시 테이블 deprecate

4) 제품 분리
   - Platform (SaaS 본사): 빌링, 권한, 채널 커넥터, 인프라
   - Operator (공급자): 상품·가격·재고·예약·정산
   - 현재 회사 운영 기능(차량, 가이드 출근, 내부 경비 등)은
     Operator add-on 또는 별 모듈로 분리 (모든 SaaS 고객에 강제하지 않음)

한 줄:
  "지금 시스템을 고치는 것"이 아니라
  "tenant-aware Commerce Core를 만들고, Kovegas를 첫 Operator로 이전"한다.

1. SaaS 목표 아키텍처
================================================================================

1.1 액터
--------------------------------------------------------------------------------
Platform Admin     SaaS 운영자 (당신/본사)
Operator           투어 공급사 (Kovegas, 다른 Las Vegas/다른 도시 업체)
Operator User      Operator 직원 (owner / admin / ops / finance / guide…)
Channel Partner    OTA (Klook, GYG…) — Operator별로 연결
End Customer       여행객 — Operator의 Direct Web 또는 OTA경유

1.2 테넌시 모델 (권장: Shared DB)
--------------------------------------------------------------------------------

                    ┌─────────────────────────────┐
                    │         PLATFORM            │
                    │  billing · auth · connectors│
                    │  feature flags · audit      │
                    └──────────────┬──────────────┘
                                   │
              ┌────────────────────┼────────────────────┐
              ▼                    ▼                    ▼
        Operator A           Operator B           Operator C
        (Kovegas)            (Other Co)           (...)
        tenant_id=T1         tenant_id=T2
              │                    │
     products/offers/        own isolated rows
     prices/inventory/       via tenant_id + RLS
     bookings

왜 Shared DB + tenant_id 인가
- 수백~수천 Operator까지 운영 비용·마이그레이션·쿼리 단순
- Supabase/Postgres RLS와 잘 맞음
- 나중에 초대형 고객만 dedicated schema/DB로 분리 가능 (예외 경로)

쓰지 말 것 (초기)
- DB-per-tenant: 운영 악몽, 마이그레이션·커넥터 배포 비용 폭증
- 테이블마다 다른 격리 방식 혼용


1.3 모든 비즈니스 테이블의 공통 컬럼
--------------------------------------------------------------------------------
operator_id      UUID NOT NULL   -- tenant
created_at / updated_at
created_by / updated_by (optional)
deleted_at       (soft delete)

유니크 제약 예시
  UNIQUE (operator_id, product_code)
  UNIQUE (operator_id, channel_id, external_sku)  -- mapping
절대 글로벌 UNIQUE(product_code) 금지


1.4 Platform vs Operator 데이터 소유
--------------------------------------------------------------------------------
PLATFORM (tenant 없음 또는 platform_id)
  operators
  operator_subscriptions / plans / invoices
  platform_users (optional)
  channel_connector_definitions   -- Viator adapter metadata (코드+설정 스키마)
  feature_flags
  audit_logs (platform scope)

OPERATOR-SCOPED
  users / memberships / roles
  products, product_options, choices, offers
  rate_plans, price_rules, overrides, promotions
  resources, allotments, holds
  channels (operator's connected channels)
  channel_listings, external_mappings
  bookings, quotes, payments (operator merchant)
  customers (operator CRM; 이메일은 operator 내에서 unique)
  (optional modules) fleet, guides, expenses…


1.5 Identity & Access
--------------------------------------------------------------------------------
Auth: Supabase Auth (또는 Clerk 등) — 사용자는 글로벌
Membership: operator_members(user_id, operator_id, role)

Roles (최소)
  owner, admin, ops, finance, guide, read_only

RLS 패턴
  current_operator_id() from JWT claim / session setting
  USING (operator_id = current_operator_id())

중요
  - 직원은 여러 Operator에 속할 수 있음 (agency/멀티브랜드)
  - 요청마다 active_operator_id 필수
  - Platform admin은 bypass role


1.6 Commerce Core (v2) — tenant-aware
--------------------------------------------------------------------------------
v2 greenfield 문서의 도메인을 그대로 사용하되 전부 operator_id:

Catalog    Product / ProductOption / Choice* / Addon
Offer      OfferSet / Offer / OfferComponent
Pricing    RatePlan / PriceRule / PriceOverride / Promotion
Inventory  Resource / Allotment / Hold / Ledger
Calendar   ScheduleRule / AvailabilitySlot / StopSell
Distribution Channel / Listing / ExternalMapping / Sync*
Booking    Quote / Booking / LineItem (snapshot)
Automation PricingSuggestion (later)

엔진 API는 항상 첫 인자로 operator context:
  quote(operator_id, …)
  holdInventory(operator_id, …)
  syncOffer(operator_id, …)


1.7 Direct Web / White-label
--------------------------------------------------------------------------------
Operator마다
  - custom domain / subdomain (kovegas.com, other.com)
  - branding theme
  - checkout & payment account (Stripe Connect 권장)

Platform은
  - hosted booking widget / pages
  - Connect onboarding
  - 수수료(application fee) 옵션


1.8 OTA Connectors (플랫폼 자산)
--------------------------------------------------------------------------------
Adapter는 Platform이 소유 (코드)
Credentials·mapping은 Operator 소유 (데이터)

operators
  └── channel_connections (operator_id, platform, credentials vault ref)
        └── external_mappings
        └── sync_jobs

새 OTA 추가 = Platform에 Adapter 배포
새 공급자 추가 = Connection + Mapping만 (코드 변경 없음)

4. 마이그레이션 전략 (Kovegas → Tenant #1 → Multi-tenant)
================================================================================

원칙
--------------------------------------------------------------------------------
- 기존 예약/정산/운영을 멈추지 않는다
- 새 SaaS 코어를 옆에 만들고 dual-read/write 후 전환
- "다른 공급자 오픈"은 Kovegas가 v2 코어에서 안정화된 뒤


PHASE 0 — 결정·동결 (1~2주)
--------------------------------------------------------------------------------
산출물
  - Operator/Plan/Role 용어 확정 (ADR)
  - 모듈 경계: Commerce Core vs Operations Suite
  - calculate_dynamic_price 등 원격 RPC를 repo에 고정
  - 가격 golden dataset (상품×채널×날짜×초이스 → 금액)
  - 성공 지표: 가격 diff 0, 예약 생성 성공률, RLS 누수 0

결정 (권장 기본값)
  - tenancy = shared DB + operator_id
  - PK = UUID
  - 결제 = Stripe Connect Express/Standard
  - 멀티브랜드 = membership multi-operator


PHASE 1 — Tenant Foundation (무중단)
--------------------------------------------------------------------------------
작업
  1. operators 테이블 생성
  2. Kovegas row 생성 (T_KOVEgAS)
  3. 핵심 테이블에 operator_id NULL 허용으로 추가
  4. backfill: 모든 기존 행 operator_id = T_KOVEgAS
  5. NOT NULL + 인덱스 + (operator_id, …) unique 재정의
  6. operator_members: 기존 team 이메일을 membership으로 매핑
  7. RLS: current_operator_id() 도입 (초기엔 Kovegas 고정 claim)
  8. 앱 컨텍스트에 activeOperatorId 주입

성공 조건
  - 기존 UI/예약 100% 동작
  - 쿼리에 operator_id 필터가 생기기 시작
  - 다른 tenant 행이 아직 없어도 격리 코드 경로 존재


PHASE 2 — Commerce Core v2 (신설, dual-write)
--------------------------------------------------------------------------------
작업
  1. v2 테이블 생성 (모두 operator_id 포함)
  2. Price Engine / Inventory Engine / Quote API
  3. 관리자 저장 시:
       legacy dynamic_pricing 기록 + v2 rules/offers/overrides 기록
  4. Backfill 스크립트 (Kovegas 데이터 → v2)
  5. Shadow quote: 구 RPC vs 신 Engine 금액 비교 로그
  6. feature flag로 상품 단위 cutover

성공 조건
  - shadow diff ≈ 0 (허용 오차 정의)
  - 홈페이지 checkout이 flag on 상품에서 v2 quote 사용


PHASE 3 — Booking Snapshot & Inventory
--------------------------------------------------------------------------------
작업
  1. Quote → Hold → Booking commit 오케스트레이션
  2. reservation_* 를 v2 booking_* 로 매핑 또는
     기존 reservations를 booking facade 뒤로 숨김
  3. 새 예약부터 offer_id / money_breakdown_json / inventory_sale_id 저장
  4. 과거 예약은 스냅샷 그대로 유지 (강제 변환 금지)

성공 조건
  - 신규 예약 읽기/정산/취소 정상
  - 과거 예약 조회·정산 회귀 없음


PHASE 4 — Distribution & First OTA Adapter
--------------------------------------------------------------------------------
작업
  1. channel_connections + external_mappings
  2. outbox sync_events
  3. Adapter 1개 (Viator 또는 Klook 중 실제 계약 있는 것)
  4. Mapping admin UI

성공 조건
  - 1개 OTA에 availability/rate push 또는 booking pull 중 하나 E2E


PHASE 5 — SaaS Onboarding (두 번째 Operator)
--------------------------------------------------------------------------------
작업
  1. Sign-up / Create Operator wizard
  2. Invite members
  3. Stripe Connect onboarding
  4. Subdomain/custom domain
  5. Plan limits (products, users, channels)
  6. 내부 테스트용 "Operator B" 생성 후 완전 격리 검증
  7. 첫 외부 공급자 파일럿 (제한 기능)

성공 조건
  - Operator B가 A 데이터를 어떤 API/UI로도 볼 수 없음 (침투 테스트)
  - 파일럿 공급자가 상품 등록→가격→예약 가능


PHASE 6 — Modularize Operations Suite
--------------------------------------------------------------------------------
작업
  - 차량/가이드/내부 경비/팀 도구를 operator module로 분리
  - Commerce-only 고객은 이 모듈 없이 사용
  - Kovegas는 모듈 ON


PHASE 7 — Decommission Legacy
--------------------------------------------------------------------------------
작업
  - dynamic_pricing JSON 경로 제거
  - 구 RPC 제거
  - 문서/어드민 레거시 화면 제거
  - backup retention 정책

Backfill 규칙
--------------------------------------------------------------------------------
1. 모든 행에 operator_id = Kovegas

3. absolute 소조합 → Offer + PriceOverride
4. is_sale_available=false → StopSell / slot closed
5. 예약 금액 재계산 금지 (스냅샷 보존)
6. OTA external ids는 매핑 테이블로만 (추측 자동확정 금지)
